# 差异分析

## 结论

当前项目与“一期企业内部知识库底座”的核心能力差距不大，主要问题不在于缺少功能，而在于产品表达、角色边界、用户主流程和运营治理需要继续收敛。

后续优化应优先解决用户是否能顺畅完成业务目标，而不是继续扩展技术能力。

## 当前已具备能力

- 文档型知识库和 FAQ 型知识库。
- 文件上传、文件夹导入、URL 导入、在线录入。
- 文档预览、重解析、标签管理。
- BM25、向量检索、混合检索、Rerank。
- 多轮问答、建议问题、引用来源。
- OIDC、JWT、API Key、多租户、共享空间。
- 私有化部署、对象存储、向量库、解析服务。
- Agent、MCP、Web Search、知识图谱等扩展能力。

## 主要差异

### 产品层

- 前台仍可能暴露过多 AI 平台能力，容易稀释“企业知识库底座”的一期定位。
- Agent、MCP、Web Search、GraphRAG、高级模型参数不应干扰普通租户主流程。
- 首页、设置、知识库、FAQ、问答需要持续围绕用户任务收敛。

### 用户体验层

- 普通租户首次进入后，需要更明确地知道“先建知识库还是先提问”。
- 导入、解析、失败、无结果、无来源等状态需要更强的下一步提示。
- 问答结果需要突出“来源是什么、为什么可信、不能回答怎么办”。

### 角色层

- 当前项目应统一为两个角色：平台超级管理员和普通租户。
- “租户管理员”只是普通租户的称呼，不作为独立角色设计。
- 不再使用普通用户、编辑者、空间管理员等多角色模型作为一期产品依据。

### 实施层

- 知识分类、标签规范、FAQ 模板和试点问题样本集需要与业务方一起沉淀。
- 知识维护责任人、更新机制、失效治理还需要形成运营闭环。

### 运维层

- 生产上线前仍需确认备份、监控、告警、恢复流程。
- 模型、检索、切块、Rerank 等默认参数需要以平台配置方式收敛。

## 差异判断

| 领域 | 差异程度 | 判断 |
|---|---|---|
| 登录与基础权限 | 低 | 需要重点核对两角色边界 |
| 知识库与 FAQ | 低 | 主能力已具备，继续优化体验 |
| 搜索与问答 | 低到中 | 需要强化可信度和失败反馈 |
| 页面与导航表达 | 中 | 继续收敛普通租户主路径 |
| 运营治理 | 中 | 需要试点数据和复盘机制 |
| 运维上线 | 中 | 需要上线前检查和演练 |
