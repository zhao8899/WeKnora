# 知识智能全链路设计文档

**日期：** 2026-04-20  
**状态：** 已确认，待实施  
**定位：** 将 WeKnora 从"知识检索工具"升级为企业"AI Executive Assistant"，三层递进形成完整飞轮

---

## 一、背景与目标

### 核心问题

1. **知识入库成本高**：员工手动上传文件，缺乏自动从网站、API、企业系统拉取的能力
2. **答案可信度不透明**：用户无法判断 AI 回答是否准确，缺少置信度和溯源信号，影响采纳率
3. **管理员无法度量价值**：看不到哪些知识被频繁引用、哪些文档从未被检索、知识库有哪些覆盖空白

### 设计定位（Karpathy EA 思路）

把系统定位为组织的 **EA（Executive Assistant）**：它能访问所有组织知识，在用户提问时跨源综合回答，并能主动告知"我在这方面的知识有限"。

### 三层架构目标

```
[入库层] 自动拉取 + 智能摘要  →  知识库持续"活"起来
     ↓
[检索层] 多维置信度 + 溯源 + 反馈  →  用户信任每一个回答
     ↓
[洞察层] 知识健康看板  →  管理员看清 EA 的知识边界
```

## 二、整体架构

```
┌─────────────────────────────────────────────────────────┐
│                  WeKnora 知识智能全链路                    │
├─────────────────────────────────────────────────────────┤
│  [入库层] KnowledgeConnector Framework                   │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐  │
│  │ Web/RSS  │ │ REST API │ │  飞书文档 │ │ Notion等 │  │
│  └────┬─────┘ └────┬─────┘ └────┬─────┘ └────┬─────┘  │
│       └────────────┴────────────┴─────────────┘         │
│                    ConnectorJob（调度 + 变更检测）          │
│                    自动摘要生成 + 实体标签提取               │
├─────────────────────────────────────────────────────────┤
│  [检索层] ConfidenceEngine                               │
│  多维评分 → 综合置信度 → 前端展示 → 用户反馈 → 权重更新       │
├─────────────────────────────────────────────────────────┤
│  [洞察层] KnowledgeHealthDashboard                       │
│  热点问题 │ 覆盖空白 │ 陈旧文档预警 │ 引用热力图              │
└─────────────────────────────────────────────────────────┘
```

**关键设计原则：**
- 三层数据流单向：入库层 → 检索层，反馈数据异步流回权重表，洞察层只读不写
- 连接器插件化：新增 connector 不影响主流程
- 反馈异步化：用户反馈影响来源权重，不触发向量索引重建

## 三、入库层设计

### 3.1 统一 Connector 接口

```go
type Connector interface {
    Meta() ConnectorMeta
    Validate(ctx context.Context, cfg Config) error
    Fetch(ctx context.Context, cfg Config, since time.Time) ([]RawDocument, error)
    SupportedContentTypes() []string
}

type RawDocument struct {
    ExternalID  string
    Title       string
    Content     string
    ContentType string
    SourceURL   string
    PublishedAt time.Time
    Metadata    map[string]string
}
```

### 3.2 三类连接器

#### Web 连接器
- **输入**：单页 URL / 站点地图 URL / RSS Feed
- **抓取策略**：Readability 正文提取，站点地图展开按 lastmod 增量同步
- **变更检测**：ETag / Last-Modified / 内容 Hash 三级对比

#### REST API 连接器（可配置，无需写代码）
- **配置项**：Endpoint、认证方式（Bearer/API Key/OAuth2）、字段映射、分页模式
- **分页支持**：page-number / cursor / link-header
- **适用场景**：企业自建系统、钉钉开放平台、语雀 API

#### 企业系统连接器（插件式）
- 已有：飞书文档（`internal/datasource/connector/feishu`）
- 待实现：Notion、钉钉知识库、Confluence

### 3.3 调度与变更检测

**触发方式：**
- 手动触发（管理员立即同步）
- 定时调度（Cron，默认每 6 小时）
- Webhook（企业系统推送变更通知）

**增量同步逻辑：**
```
Fetch(ctx, cfg, lastSyncAt)
  → 对比 ExternalID + ContentHash
  → 新增：入库建索引
  → 变更：更新原文 + 重新向量化 + freshness 重置
  → 消失：标记 archived（保留引用历史，不物理删除）
```

**入库后自动处理：**
- 调用 LLM 生成文档摘要（写入 `knowledge.summary`）
- 自动提取实体标签（人名、产品名、部门名）

### 3.4 前端配置入口

位置：知识库设置 → 数据来源 → 添加连接器

显示信息：连接器类型、同步频率、上次同步时间、文档数量、状态、最近同步日志（可展开）

## 四、检索层设计

### 4.1 置信度评分模型

```
ConfidenceScore = w1×VectorSim + w2×KeywordHit + w3×SourceWeight + w4×Freshness

默认权重：
  w1 = 0.40  向量相似度（语义匹配质量）
  w2 = 0.25  关键词命中（精确匹配补充）
  w3 = 0.20  来源权重（人工评分 + 用户反馈积累）
  w4 = 0.15  时效性（文档修改时间衰减函数）
```

**多源综合加成：**
- 2+ 个来源结论一致 → 置信度 +10%，上限 100%
- 来源间存在矛盾 → 主动标注"存在不同说法"，综合置信度不加成
- 来源均为高权重文档（官方制度、技术规范）→ 额外 +5%

### 4.2 前端展示策略

| 置信度区间 | 颜色 | 展示文案 |
|-----------|------|---------|
| 85–100% | 绿色 | 高度可信，来自权威文档 |
| 60–84% | 蓝色 | 可参考，建议核对来源 |
| 40–59% | 橙色 | 仅供参考，知识库覆盖有限 |
| < 40% | 灰色 | 知识库中未找到可靠依据（主动声明） |

**展示内容：**
- 置信度进度条 + 百分比
- "综合了 N 个来源"
- 引用来源列表（文档名 + 页码/段落，可点击高亮原文）
- 反馈按钮：[👍 准确] [✏️ 有误，我来纠正] [❓ 来源看不懂]

### 4.3 反馈闭环机制

**用户操作流程：**
```
点击"有误，我来纠正"
  → 弹出轻量纠错表单（选择问题类型 + 可选补充）
  → 异步写入 source_feedback 表
  → 定时任务（每日）重新计算 source_weight
  → 下次检索权重已更新（无需重建向量索引）
```

**反馈类型与权重影响：**

| 反馈类型 | source_weight 影响 | 后续动作 |
|---------|-------------------|---------|
| 答案准确 | +0.02 | 无 |
| 部分正确 | 0 | 记录待审核 |
| 答案有误 | -0.05 | 触发管理员审核通知 |
| 文档已过期 | 0 | 标记 freshness_flag，触发重新拉取 |

### 4.4 新增后端接口

```
GET  /api/chat/answer/{id}/confidence    返回置信度详情
POST /api/chat/answer/{id}/feedback      提交用户反馈
GET  /api/source/{id}/weight-history     来源权重变化历史（管理员）
```

## 五、洞察层设计

### 5.1 四个核心指标

| 指标 | 数据来源 | 用途 |
|------|---------|------|
| 热点问题 Top20 | session 表用户提问聚类 | 判断是否有对应知识覆盖 |
| 覆盖空白 | confidence < 0.4 的问答记录 | 提示管理员补充知识 |
| 陈旧文档预警 | knowledge.updated_at 超过 90 天 | 提示重审或触发重新拉取 |
| 引用热力图 | document_access_log 检索命中记录 | 识别从未被引用的文档，建议归档 |

### 5.2 前端看板位置

- 入口：设置 → 知识健康（管理员可见）
- 数据刷新：每日计算，页面展示最新计算结果

### 5.3 新增后端接口

```
GET /api/analytics/hot-questions         热点问题列表
GET /api/analytics/coverage-gaps         覆盖空白问题列表
GET /api/analytics/stale-documents       陈旧文档列表
GET /api/analytics/document-heatmap      文档引用热力图数据
```

## 七、关键设计决策记录

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 置信度计算时机 | 查询时动态计算 | 支持权重热更新，无需重建索引 |
| 反馈写回方式 | 异步 + 每日批量 | 避免用户操作影响检索实时性能 |
| 消失文档处理 | 标记 archived，不物理删除 | 保留历史引用，避免已有对话中断 |
| 连接器扩展方式 | interface + 插件注册 | 新增 connector 不修改主流程 |
| 看板数据刷新 | 每日批量计算 | 数据量大时实时计算代价过高 |
